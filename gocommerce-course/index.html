Module 1

The Checkout Journey

Have you ever wondered what happens under the hood when a user clicks "Buy" on a static website? Static sites can't securely process payments or manage inventory on their own.

That's where GoCommerce steps in. Think of GoCommerce as a highly efficient checkout clerk at a specialized boutique. The clerk coordinates between the customer, the bank, and the stockroom without ever moving from the register.

The Bridge

GoCommerce is an API that bridges the gap between static pages (like a Gatsby or Next.js site) and dynamic backend services (like databases and payment gateways).

Let's look at the main structure that holds this all together:

api/api.go
type API struct {
	handler http.Handler
	db      *gorm.DB
	config  *conf.Configuration
	mailer  mailer.Mailer
	version string
	ext     *extensions
}
1 The Router: handler manages all the incoming web requests.
2 The Storage: db connects to the database to save orders and items.
3 The Postman: mailer sends out order receipts and notifications.

Understanding this flow makes debugging "failed to charge" or "didn't receive email" bugs much easier. Here's exactly how an order travels through the system.

The Power of Context

Because middleware injects context, when building new features you don't need to manually pass the database connection around. You simply grab it from the request context!

Let's look at exactly how the middleware injects this context:

api/router.go
func (m middlewareHandler) serve(next http.Handler, w http.ResponseWriter, r *http.Request) {
	ctx, err := m(w, r)
	if err != nil {
		handleError(err, w, r)
		return
	}
	if ctx != nil {
		r = r.WithContext(ctx)
	}
	next.ServeHTTP(w, r)
}
1 Run the middleware: First, we run the custom middleware logic. If it fails (like a bad token), we handle the error immediately and stop.
2 Attach Context: If the middleware returns a new context (like a database connection), we attach it to the original request using WithContext.
3 Pass the baton: Finally, we call next.ServeHTTP to pass the request to the next handler in the chain.
Module 3

Managing the Cart

An order is more than just a total amount; it's a collection of addresses, products, and calculations. Think of it as a detailed filing cabinet where every folder (Order) has specific sheets inside.

In GoCommerce, these entities are represented in the models/ directory.

models/ Core data structures
order.go The main folder holding the cart
line_item.go Individual products inside the order
address.go Where the order is going

Let's look at the Order struct, which defines exactly how an order is saved and sent as JSON:

models/order.go
type Order struct {
	ID              string       `json:"id"`
	LineItems       []LineItem   `json:"line_items"`
	SubTotal        uint64       `json:"sub_total"`
	Taxes           uint64       `json:"taxes"`
	Total           uint64       `json:"total"`
	Email           string       `json:"email"`
	ShippingAddress *Address     `json:"shipping_address,omitempty"`
	BillingAddress  *Address     `json:"billing_address,omitempty"`
}
1 The Products: LineItems is an array of the individual items purchased.
2 The Math: Totals and taxes are stored as uint64 (unsigned 64-bit integers) representing cents, to avoid floating point math errors!
3 The Locations: Pointers to Address structs. The omitempty tag means if they aren't provided, they won't show up in the JSON response at all.

Module 4

Taking the Money

An order is just a wishlist until the money actually moves. But managing different payment gateways can turn your code into a messy knot of if/else statements.

GoCommerce solves this by using a universal credit card terminal metaphor. It defines an interface that any payment provider must follow.

payments/payments.go
type Provider interface {
	Charge(*models.Order, *models.Transaction) error
	Refund(*models.Transaction) error
	Name() string
}
1 Take Money: Any provider must be able to charge an Order and update the Transaction.
2 Give Money Back: They must also support refunding a specific transaction.
3 Identify: And they must be able to tell GoCommerce their name (e.g., "stripe").

Because of this interface, GoCommerce can easily swap or support multiple providers:

Module 5

Delivering the Goods

How do you securely sell a PDF or eBook if your site is just static HTML? You can't just put a public link to the file on your site!

GoCommerce acts as a secure vault. It uses JWTs as temporary, verifiable tickets that grant access to downloads.

1
Customer Buys Product

The payment succeeds and the order is marked paid.

2
GoCommerce Mints a Token

A JWT is created specifically for that file and that user.

3
User Clicks Link

The email receipt contains a URL with the JWT embedded in it.

4
Validation & Delivery

GoCommerce verifies the token signature. If valid, it serves the file.

Here is the exact code where GoCommerce "mints" that ticket:

api/download.go
claims := jwt.StandardClaims{
	Id:        dl.ID,
	Subject:   "download",
	ExpiresAt: time.Now().Add(time.Hour * 24).Unix(),
}
token := jwt.NewWithClaims(jwt.SigningMethodHS256, claims)

tokenString, err := token.SignedString([]byte(config.JWT.Secret))
1 The Claims: The token clearly states what it's for (the specific download ID) and when it expires (24 hours from now).
2 The Method: We choose HS256, a fast and secure algorithm for signing tokens.
3 The Signature: We sign the ticket using our secret configuration key. If anyone tampers with the token, the signature will be invalid!
Module 6

Closing the Loop

The transaction is paid, the digital ticket is minted, and the data is saved in the database. But the customer doesn't know any of that yet!

We need a post-event wrap-up where the organizer sends out thank you notes (Emails) and tells the external systems that the order is complete (Webhooks).

Asynchronous by default

GoCommerce tries to send emails without blocking the main web request. You don't want a user waiting for the page to load just because the SMTP server is slow!

Here is how the Mailer prepares to send the confirmation receipt to the customer:

mailer/mailer.go
func (m *mailer) OrderConfirmation(order *models.Order) error {
	m.log.Debugf("Sending order confirmation for %v", order.ID)

	subject := m.config.Subjects.OrderConfirmation
	if subject == "" {
		subject = "Order Confirmation"
	}

	return m.sendEmail(order.Email, subject, "order_confirmation", order)
}
1 Log the attempt: Always log before an external action so you can trace errors later.
2 Dynamic Subject: It checks the configuration to see if the merchant set a custom subject line. If not, it falls back to a default.
3 Send it off: It passes the customer's email, the subject, the name of the HTML template to use, and the full Order object so the template can render the line items.

Congratulations! You've just followed an order all the way from the user's browser, through the GoCommerce API, into the database, through the payment processor, and finally back out as an email receipt.
